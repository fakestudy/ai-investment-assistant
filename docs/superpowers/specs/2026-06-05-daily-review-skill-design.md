# Daily Review Skill 设计

## 目标

创建项目级 `daily-review` skill。它基于当前项目当天的 Codex 会话、Git 提交与未提交改动，以资深全栈工程师视角进行逐题面试，评估用户的技术理解与独立思考能力，并生成可持续追踪的复盘报告。

## 范围

- 只分析当前 Git 项目。
- 只读取 Codex 本地会话，且会话 `cwd` 必须属于当前项目。
- 以 `Asia/Shanghai` 的自然日划分“当天”。
- 读取当前分支当天的 Git 提交及提交 diff。
- 读取当前工作区未提交改动，但未提交改动没有可靠时间戳，只能作为辅助出题证据，不能证明当天完成度；未跟踪内容需限制体积并排除敏感文件与符号链接。
- 读取上一份复盘报告，检查其中 TODO 的巩固情况。
- 不自动提交复盘报告。
- 第一版不支持其他 Code Agent、远端分支或自动定时执行。

## 文件结构

```text
skills/daily-review/
├── SKILL.md
├── agents/openai.yaml
└── scripts/
    └── collect_context.py
```

复盘报告写入：

```text
daily-reviews/NN-YYYY-MM-DD.md
```

`NN` 为全局递增序号。同一天重复执行时更新已有报告，不生成新序号。

## 上下文采集

`collect_context.py` 接收项目路径，输出结构化 JSON，包含：

- 项目根目录、分支、日期与时区
- 当天当前项目的 Codex 用户和助手会话正文
- 当前分支当天提交的元数据及 diff
- 当前未提交改动
- 最新一份历史复盘及 TODO
- 当天已有复盘文件与下一可用序号

采集时排除工具调用原始输出、系统提示、其他项目会话和远端分支提交。无法取得任何有效证据时，skill 停止面试和评分，并明确说明“今日无可复盘上下文”。

## 面试流程

1. 运行采集脚本并阅读证据。
2. 提炼当天真正涉及的技术主题、决策和实现。
3. 组织 5–8 个核心问题，每次只问一个。
4. 根据回答决定是否追问，每个核心问题最多追问 2 次。
5. 证据已经足以判断理解程度时，可以提前结束，不为凑数量继续提问。
6. 完成面试后评分并生成报告。

问题优先检验：

- 实现原理和问题根因
- 方案取舍及被放弃方案
- 边界条件和失败场景
- 将当天知识迁移到新问题的能力
- 上一份 TODO 是否真正理解和掌握

回答含糊、只复述代码或与证据冲突时必须追问，不替用户补全推理。

## 评分

总分 100：

| 维度 | 分值 |
| --- | ---: |
| 技术理解 | 40 |
| 独立思考 | 30 |
| 推理与表达 | 15 |
| 知识迁移 | 10 |
| 学习闭环 | 5 |

评分只依据用户在面试中的回答。会话、提交和代码改动用于出题与核验，不能直接替代理解。每个扣分项必须关联具体回答或缺失点，禁止使用代码量、提交数量或主观努力程度加分。

## 报告

报告包含：

- 今日证据摘要
- 问题、回答与追问结论
- 各维度评分及扣分依据
- 总分与客观评价
- 已掌握内容
- 薄弱概念与错误认知
- 上一份 TODO 完成情况
- 下一步学习 TODO

TODO 必须具体、可执行、可验证。例如：“脱离代码解释 SSE 断线续传流程，并指出 3 个失败边界”，而不是“继续学习 SSE”。

## 验证

- 使用临时项目和伪造 Codex JSONL 会话测试日期、`cwd`、角色正文和噪声过滤。
- 测试有提交、只有未提交改动、只有会话、完全无证据四种情况。
- 测试复盘序号递增和同一天报告复用。
- 使用 skill 校验脚本检查目录及 frontmatter。
- 用一次真实当天上下文进行前向测试，确认它严格逐题提问，不提前评分或一次抛出多个问题。
