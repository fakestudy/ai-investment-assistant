# ChatGPT-like Chat Only Design

## 1. 背景

本设计基于 `docs/project/PLAN_CHAT.md`，本期只建设通用 AI 聊天助手，不实现投资业务能力。聊天助手作为未来投资助手的工程基座，优先跑通多会话、流式输出、Agent 工具事件、消息持久化和 ChatGPT-like 桌面端体验。

当前前端已有 Next.js、ai-elements、shadcn/ui 等依赖基础，但首页仍是默认模板；后端目录尚未落地。因此本设计同时约束前端体验边界和前后端职责边界，避免把范围扩展到文件上传、模型管理、登录、多端适配等非本期目标。

## 2. 目标

- 做一个桌面端优先的 ChatGPT-like 通用聊天界面。
- 保持 ChatGPT 默认浅色主题风格：中性色、低干扰、圆角输入框、左侧会话栏、居中消息阅读区。
- 支持多会话持久化、流式回复、Markdown/Code 渲染、思考过程展示、详细工具调用展示。
- 支持 `web_search` 和 `fetch_url` 两个后端工具的前端过程呈现。
- 保持单用户、本地使用、无登录、无模型切换、无文件上传。

## 3. 非目标

- 不做投资助手业务功能。
- 不做文件、图片、附件上传和解析。
- 不做模型切换 UI。
- 不做登录、注册、多用户、权限。
- 不做移动端适配。
- 不做会话搜索。
- 不做消息分支树或回答版本切换。
- 不做分享会话、归档、语音输入、Canvas、GPTs、Projects。
- 不做暗色主题；第一版仅浅色。

## 4. 用户体验范围

### 4.1 页面布局

桌面端采用 ChatGPT-like 三段式布局：

- 左侧会话栏：新建会话、会话列表、重命名、删除。
- 中间消息区：显示用户消息、助手消息、工具详情、错误状态。
- 底部输入区：多行输入、发送按钮、停止生成状态。

移动端暂不适配。若窄屏访问，只保证不阻塞基础使用，不作为验收目标。

### 4.2 主题风格

第一版只做浅色主题：

- 背景以白色和浅灰为主。
- 左侧栏使用浅灰底，当前会话用白色或更明显的浅色高亮。
- 输入框使用大圆角容器，固定在底部阅读区。
- 消息区保持宽度收敛，避免全屏横向铺开。
- 不引入投资助手专属品牌色或复杂视觉语言。

### 4.3 空状态

新会话空状态只展示简洁欢迎语和底部输入框，不做建议问题卡片。这样保留 ChatGPT 的简洁感，同时避免为建议 prompt 设计额外内容策略。

## 5. 功能清单

### 5.1 会话管理

必须支持：

- 新建会话。
- 切换会话。
- 重命名会话。
- 删除会话。
- 第一轮对话后自动生成短标题并更新左侧栏。

暂不支持：

- 会话搜索。
- 会话归档。
- 会话分享。
- 多用户维度的会话隔离。

### 5.2 输入能力

必须支持：

- 多行文本输入。
- `Enter` 发送。
- `Shift+Enter` 换行。
- 发送中禁用重复提交。
- 生成中展示停止按钮。
- 用户可停止当前流式生成。

不展示附件入口，避免暗示文件上传能力。

### 5.3 消息展示

必须支持：

- 用户消息和助手消息的基础排版。
- Assistant SSE 增量输出。
- Markdown 渲染。
- 代码块高亮。
- 复制代码。
- 复制助手回答。
- 重新生成最后一次助手回答。
- 点赞和点踩反馈入口。
- 错误消息展示。

### 5.4 历史消息编辑

允许编辑用户历史消息。编辑后采用简单线性模型：

- 保存编辑后的用户消息。
- 删除或废弃该消息之后的旧消息。
- 基于编辑后的上下文重新生成后续助手回答。

不做 ChatGPT 式分支版本切换，也不做完整分支树。

### 5.5 思考过程

后端 SSE 若输出 reasoning 事件，前端应展示为可折叠区域。默认策略：

- 生成中可显示“正在思考”状态。
- 完成后可折叠查看。
- 若模型或后端不返回 reasoning，则不展示空模块。

### 5.6 工具调用展示

后端负责执行 `web_search` 和 `fetch_url`。前端只消费 SSE 工具事件并展示：

- 工具名。
- 调用参数。
- 调用状态。
- 耗时。
- 结果摘要。
- 错误信息。
- 引用链接。

工具过程采用详细展示，偏开发者学习和调试视角。为避免干扰正文，工具详情应以独立块展示，后续可升级为可折叠。

## 6. 前后端职责

### 6.1 前端职责

- 实现 ChatGPT-like 桌面端浅色 UI。
- 调用后端 REST API 管理会话和消息。
- 消费后端 SSE 事件流。
- 将 `delta`、`reasoning`、`tool_call`、`tool_result`、`done`、`error` 映射为 UI 状态。
- 渲染 Markdown、代码块、工具详情、错误状态。
- 处理停止生成、重新生成、复制、反馈、编辑消息等交互。

### 6.2 后端职责

- 提供会话 CRUD API。
- 提供消息持久化和历史读取。
- 接入 DeepSeek。
- 通过 Eino 编排 ReAct Agent 和 ToolNode。
- 实现 `web_search` 和 `fetch_url`。
- 生成会话标题。
- 输出稳定的 SSE 协议。
- 处理模型错误、工具错误、超时、取消、重试和持久化一致性。

## 7. 数据模型

沿用 `PLAN_CHAT.md` 的 v0.1 草案：

- `conversations`：`id`、`title`、`created_at`、`updated_at`。
- `messages`：`id`、`conversation_id`、`role`、`content`、`tool_calls`、`reasoning`、`created_at`。
- `tool_invocations`：`id`、`message_id`、`tool_name`、`args`、`result`、`latency_ms`、`status`。

第一版不引入 `users`、`attachments`、`embeddings`。

## 8. SSE 事件

后端至少输出以下事件类型：

- `delta`：助手文本增量。
- `reasoning`：思考过程增量。
- `tool_call`：工具调用开始或参数更新。
- `tool_result`：工具结果或工具错误。
- `done`：本次生成完成。
- `error`：模型、网络、工具或服务端错误。

前端必须能处理事件乱序之外的常见异常：连接中断、用户取消、服务端错误、工具失败、空响应。

## 9. 错误处理

用户可见错误必须明确且可恢复：

- SSE 中断：提示连接中断，并允许重新生成。
- 模型失败：显示失败原因摘要，并允许重试。
- 工具失败：工具详情块中展示失败状态，不吞掉错误。
- 用户取消：停止继续追加 token，并保留已生成内容或按后端返回状态收束。
- 删除会话：需要确认，避免误删。

## 10. 验收标准

- 打开页面后能看到 ChatGPT-like 浅色桌面布局。
- 能新建会话、发送消息、流式接收助手回复。
- 刷新页面后，会话和历史消息仍存在。
- 第一轮对话后左侧栏标题自动更新。
- 能切换、重命名、删除会话。
- Markdown 和代码块展示正常，代码可复制。
- 能停止生成、重新生成最后一次回答。
- 能编辑用户历史消息，并基于编辑后的内容重新生成。
- 工具调用过程能详细展示工具名、参数、耗时、结果摘要、错误和引用。
- 不出现文件上传、模型切换、登录、移动端专项入口。

## 11. 推荐实施顺序

1. 实现静态 ChatGPT-like 桌面布局。
2. 接入会话 CRUD 和消息持久化。
3. 接入 SSE 流式聊天。
4. 完成停止生成、错误状态和重新生成。
5. 完成 Markdown、代码高亮、复制、反馈、编辑消息。
6. 接入 reasoning 和工具事件展示。
7. 打磨自动标题、删除确认、加载态和空状态。
